# 报告1：阶段进展（按 report000 结构）

日期：2026-04-04

## 1. 当前研究目的（对应 report000）

本阶段聚焦 `report000` 的阶段A目标：  
在固定六视角数据上稳定“人脸检测 -> 部位定位 -> mask/分割条件构建”链路，为后续 `LoRA + ControlNet + Inpainting` 提供可靠输入条件。

更具体地说，这一阶段不是训练最终生成模型，而是在解决生成前的基础条件问题：

1. 哪个检测器在六视角上最稳。  
2. 鼻子、眼睛、嘴巴等关键部位能否稳定定位。  
3. SAM 能否基于这些提示框生成可用的局部 mask。  
4. 输出是否能保持目录树一致、便于后续批处理与训练集构建。  

## 2. 本阶段已经做了什么

本阶段已完成的工程工作如下：

1. 建立统一入口 `face_detection_batch.py`，把流程拆成三个独立模式：
   - `detect_compare`：多检测器横向对比
   - `eye_mask`：眼部打码
   - `sam_mask`：基于检测结果生成 SAM mask
2. 完成多检测器统一后端接口，支持不同模型共用一套输入输出逻辑。  
3. 完成批处理目录遍历、自然排序、保持原目录树输出。  
4. 完成检测结果可视化，包括：
   - 人脸框
   - 关键点
   - 语义部位框（眼、鼻、口）
   - 可选分割预览
5. 完成 SAM 条件构建与部位 mask 输出（`face / left_eye / right_eye / nose / mouth`）。  
6. 完成六视角参数化机制，支持按 `1~6` 视角单独做 ratio 缩放和偏移。  
7. 完成基础统计输出，包括耗时、漏检、读取失败、推理失败。  

## 3. 当前代码成果落点

当前仓库已经形成了可复用的前处理子系统，主要对应：

1. 检测对比：`face_detection/batch.py`  
2. 打码流程：`face_detection/masking.py`  
3. 分割流程：`face_detection/sam_mask.py`  
4. 分割实现：`face_detection/segmentation.py`  
5. 检测器后端：`face_detection/backends.py`  
6. 参数与路径配置：`face_detection/settings.py`  

这意味着：  
当前代码已经覆盖了“检测、提示框、mask、目录输出、统计”这一整层，后续可直接服务于 LoRA/ControlNet 训练集与推理链路构建。

## 4. 已做尝试（按实际迭代顺序）

本阶段不是一开始就做多模型对比，而是从一个非常具体的眼部遮挡任务逐步扩展出来的。实际尝试路径如下：

1. 第一版：直接做“黑条遮眼”  
   - 最早从单脚本 `mask_eyes_batch.py` 起步。  
   - 方法是用 MediaPipe Face Mesh 的眼部关键点索引，直接求双眼区域的包围框，再画黑条。  
   - 这个版本能快速跑通，但问题也很快暴露：  
     - 侧脸和底视图漏检明显  
     - 水平矩形框容易吃到鼻梁  
     - 双眼遮挡逻辑不适合“只有一个眼睛可见”的侧脸

2. 第二版：围绕眼部遮挡不断修补  
   - 先后做过这些调整：  
     - 去掉命令行参数，改成直接 `main()` 跑  
     - 修复目录遍历与自然排序  
     - 增加跳过提示、保留原目录树输出  
     - 只保留“术前/术后”等规范目录  
   - 这一阶段本质还是“为了把黑条打准”，但已经开始暴露出：  
     - 真正的瓶颈不是画框，而是前面的检测与定位质量

3. 第三版：尝试用 MTCNN 给侧脸兜底  
   - 由于 MediaPipe 在大侧脸、单眼场景下经常漏检，曾接入 `facenet-pytorch` 的 `MTCNN` 作为 fallback。  
   - 同时还做过一些侧脸启发式逻辑，比如：  
     - 假设固定视角的 `4/5.JPG` 更可能是侧脸  
     - 尝试只出一个眼睛框  
     - 调整 MTCNN 框大小和偏移  
   - 结论：  
     - MTCNN 的确能补一部分漏检  
     - 但定位稳定性一般，经常“框太粗、鼻子被吃掉、位置不准”  
     - 这类启发式规则鲁棒性不够，后续基本放弃作为主方案

4. 第四版：从“遮眼脚本”升级为“检测平台”  
   - 随着问题逐步明确，代码开始从单一打码脚本转成模块化结构：  
     - 检测对比  
     - 打码  
     - 分割  
   - 这一阶段做了大量基础工程修正：  
     - 兼容不同 `mediapipe` 版本（`solutions` / `tasks`）  
     - 统一模型路径到 `model/`  
     - 清理旧 BlazeFace 模型与不兼容权重  
     - 移除 YOLOv5 路线，统一保留 YOLOv8-Face  
     - 整理 README、requirements、目录结构

5. 第五版：系统对比多种人脸检测器  
   - 这一步才真正开始围绕“谁最适合作为条件构建入口”做横向实验。  
   - 主要试过：  
     - `MTCNN`
     - `RetinaFace`
     - `SCRFD`
     - `BlazeFace`
     - `MediaPipe Face Mesh / FaceLandmarker`
     - `YOLOv8-Face`
     - `CenterFace`
     - 以及未深入保留的 `YOLO5Face / DSFD / PyramidBox`
   - 这一阶段的重点已经不是“能不能检测到脸”，而是：  
     - 六视角下漏检多少  
     - 鼻部和部位框偏移是否可接受  
     - 速度能否支撑批处理  
     - 是否适合后续作为 SAM / Inpainting / ControlNet 的条件来源

6. 第六版：接入 SAM，开始做“条件构建”而不是“纯检测”  
   - 在有了较稳定的检测器后，进一步把目标切到 SAM 条件构建：  
     - 用检测框和关键点生成语义部位框  
     - 输出 `face / eyes / nose / mouth` 的分割 mask  
     - 支持按视角做 ratio 缩放和偏移  
   - 这一步的意义在于：  
     - 项目开始从“局部工具脚本”转成“为后续生成模型服务的前处理系统”

## 6. 当前阶段结论

基于当前实验结果，可以得到以下判断：

1. `SCRFD` 当前综合最优。  
理由：漏检 0，稳定性最高，鼻部定位也相对更稳。

2. `BlazeFace` 当前是最快方案。  
理由：漏检同样为 0，速度明显快于 SCRFD，但鼻部偏移略大。

3. `YOLOv8-Face` 具有较好的折中性。  
理由：速度和效果都可接受，漏检仅 1，可作为备选或对照组。

4. `MTCNN` 不适合当前默认主链路。  
理由：底视图漏检和整体偏移问题都比较明显。

5. `MediaPipe Face Mesh` 不适合当前主检测用途。  
理由：侧脸漏检过高，虽然关键点细，但入口检测阶段已经失效。

6. `CenterFace` 当前不稳定。  
理由：阈值敏感，易产生大量低质量框，不适合默认启用。

## 7. 当前推荐配置

当前推荐的工程策略如下：

1. 默认主模型：`SCRFD`。  
2. 高吞吐版本：`BlazeFace`。  
3. 备选模型：`YOLOv8-Face`。  
4. 暂不纳入默认流程：`MTCNN / Face Mesh / CenterFace / YOLO5Face`。  

如果目标是“为后续生成模型提供更可靠的鼻部条件”，当前更应该优先保 `SCRFD` 的稳定性，而不是只追求速度。

## 8. 当前问题与风险

当前仍存在以下问题：

1. 侧脸与底视图下，鼻部定位仍存在偏移。  
2. 一些模型虽然能检测到脸，但部位语义点不够稳，影响 SAM 提示质量。  
3. `RetinaFace` 依赖 TensorFlow 路线，GPU 环境配置成本高，复现性一般。  
4. 目前评价仍有主观成分，缺少统一的鼻部偏移量化指标。  

## 9. 下一步计划（对齐 report000）

1. 固化默认配置：`SCRFD + fallback + 六视角参数`。  
2. 增加鼻部偏移量化指标，把“看起来偏不偏”变成可统计数值。  
3. 在固定样本与全量样本分别复测，沉淀一份更标准的实验记录。  
4. 完成阶段A后，进入阶段B：构建 LoRA 训练数据、风格触发词与条件输入规范。
