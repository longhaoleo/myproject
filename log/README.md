# 项目日志（给 AI 看的项目记忆）

该目录的目标不是单纯记录“改了哪些文件”，而是让新的 AI 或协作者快速理解：

1. 这个项目为什么这样设计
2. 当前主链路的逻辑是什么
3. 方法是怎么逐步演化出来的
4. 哪些结论已经稳定，哪些仍然只是阶段性方案

也就是说，`log/` 更像“研究与工程决策说明”，而不是 changelog。
它默认是给 AI 看的，所以写法优先保证：

1. 逻辑完整
2. 方法清楚
3. 过程可追溯
4. 后续可继续接手

## 推荐阅读顺序

1. `baseline_report.md`
   - 看总目标、阶段划分、长期路线
   - 这是项目的冻结基线
2. `detection_report.md`
   - 看 detection 侧是怎么从“眼部打码脚本”演化成“多检测器 + SAM 条件构建”的
   - 重点是检测逻辑、模型选择思路、阶段A的实验结论
3. `generation_report.md`
   - 看 generation 侧是怎么接上 detection 产物的
   - 重点是训练逻辑、paired/unpaired 视角利用方式、推理与评估思路
4. 后续报告
   - 按时间递增阅读
   - 优先关注“为什么这么改”“当前方案解决了什么问题”“还保留了哪些升级口”

## 强约束

1. `baseline_report.md` 为当前基线文件。
2. 新阶段优先新增 `*_report.md`，记录新的阶段逻辑、尝试和结论。
3. 若目标确需调整，必须由项目负责人明确批准后，以“新增报告”方式记录变更。

## 补充说明

1. `baseline_report.md` 负责定义项目基线、方法论和日志书写约定。
2. `detection_report.md` 及后续报告应尽量记录真实的工程迭代过程，包括尝试、效果、保留原因和后续可能思路。
3. 如果某一轮改动已经明显改变了主链路逻辑，应该在对应报告里补“当前训练逻辑 / 当前推理逻辑 / 当前评估逻辑”，避免只看代码才能知道实际行为。
