# 报告000：项目基线、当前主路线与报告方法论

日期：2026-04-11

## 1. 当前状态与项目定位

这个项目现在不是“完整科研方案已定型”的状态，而是一个已经跑通主链、仍在继续迭代的方法验证阶段。

当前主线已经比较明确：

1. detection 负责定义 generation 要消费的条件
2. generation 负责在这些条件下做术前到术后的风格化局部重绘
3. 当前重点是把“小样本、少量缺视角、多视角一致性”这几个现实问题处理得尽可能稳

所以，这个项目现阶段更像：

1. 一个已可运行的实验主线
2. 一个仍保留升级空间的工程化研究骨架

而不是：

1. 所有组件都已经冻结的最终论文实现

## 2. 项目总目标

项目的目标，是构建一个针对特定医生风格的鼻整形术前预测系统。

输入：

1. 患者术前多视角照片

输出：

1. 该医生风格下的术后外观预测结果

当前默认认为这个目标必须同时满足三类要求：

1. 风格要求
   - 预测结果要体现目标医生的审美倾向和鼻部重塑风格
2. 结构要求
   - 非手术区域不能明显漂移，头面部比例和姿态应尽量连续
3. 人物要求
   - 结果仍然要像同一个人，而不是生成出“换脸感”

## 3. 当前主路线

### 3.1 detection 主路线

detection 的任务不是做最终生成，而是给 generation 提供稳定的、可复用的条件。

当前 detection 的主输出是：

1. `eye_mask` 后图片
2. `face_mask`
3. `inpaint_mask`
4. `feather_mask`
5. 部位级 mask 与可视化预览

这条路线的关键判断是：

1. generation 不应该自己再去推断“哪里该改”
2. 编辑区域应该在 detection 阶段就被稳定定义好

### 3.2 generation 主路线

当前 generation 的 v1 主线是：

1. 默认使用 `eye_mask` 后的术前图作为主输入
2. 训练时使用“同病例、同视角”的 `术前 -> 术后` 配对监督
3. 对少量缺视角样本，不直接丢弃，而是作为低权重身份样本继续利用
4. 底模固定为 `SDXL Inpainting`
5. 当前只训练 UNet LoRA
6. 推理默认先做 face crop，再映射回整图并按 `feather_mask` 回贴

这条主线解决的是：

1. 小样本如何最大化利用
2. 缺视角样本如何不浪费
3. 多视角输入下如何尽量稳住人物一致性

## 4. 当前方法论

### 4.1 为什么 detection 先定义编辑区域

因为鼻整形任务本质是局部编辑任务。  
如果 generation 侧每次推理时再临时决定编辑区域：

1. 训练和推理的条件定义会不一致
2. 多视角之间的编辑范围也更容易漂移

所以当前选择让 detection 先统一生成：

1. `face_mask`
2. `inpaint_mask`
3. `feather_mask`

### 4.2 为什么术前是条件图、术后是目标图

因为任务本身不是“重建术后图”，而是“根据术前图预测术后图”。

如果训练锚点放在术后图：

1. 模型会更接近自重建
2. 推理时输入分布会和训练时不一致

所以当前 generation 方案里：

1. 条件图来自术前
2. 目标图来自术后

### 4.3 为什么 `eye_mask` 后图片是主输入

因为当前数据链本身已经先走了隐私遮挡。

如果训练时用原图、推理时用打码图：

1. 会引入明显的输入分布偏移

所以现在默认逻辑是：

1. 有 `eye_mask` 后图片，就直接用它
2. 没有才回退到原图

### 4.4 为什么 paired 和 unpaired 都要利用

当前数据是小样本，而且少量病例会缺 1 到 2 个视角。

如果只保留 paired 视角：

1. 数据会被进一步压缩
2. 一部分人物头面部信息会直接浪费

所以现在的逻辑是：

1. paired 视角负责真正的术前到术后监督
2. unpaired 视角负责低权重身份保持

### 4.5 为什么一致性先做轻量方案

当前没有直接引入显式 identity loss，主要是因为：

1. 小样本阶段先要把主链跑稳
2. 过早堆更多损失会让问题难以定位
3. 先验证样本组织方式和条件设计更重要

所以当前一致性主要靠：

1. `paired_head_reference`
2. `paired_edit_crop`
3. `self_identity_head`
4. grouped sampling
5. `hard_identity_similarity / soft_face_similarity`

## 5. 当前主要尝试与阶段性效果

### 5.1 detection 侧

已尝试路线：

1. `MTCNN`
2. `RetinaFace`
3. `SCRFD`
4. `BlazeFace`
5. `MediaPipe Face Mesh / FaceLandmarker`
6. `YOLOv8-Face`
7. `CenterFace`

阶段性结论：

1. `SCRFD` 稳定性更好
2. `BlazeFace` 速度更快
3. `YOLOv8-Face` 是可接受折中

### 5.2 generation 侧

已完成的关键变化：

1. 从术后单图自监督，切到术前锚点配对监督
2. manifest 已能区分 paired / unpaired 视角
3. 样本已经拆成：
   - `paired_head_reference`
   - `paired_edit_crop`
   - `self_identity_head`
4. 推理和评估已经围绕病例级多视角组织输出

当前效果主要是工程效果：

1. 训练、推理、评估的输入输出逻辑已经打通
2. 小样本利用方式已经比早期方案更清晰

## 6. 当前局限

1. detection 侧在部分大侧脸/底视图仍有偏移风险
2. generation 当前仍是 v1 轻量一致性方案
3. 当前评估指标更偏工程验收，不是最终身份度量
4. 当前 LoRA recipe 是可落地方案，不等于最终最优方案

## 7. 后续可能思路

1. detection 线
   - 继续优化多视角检测稳定性
   - 继续比较 `SCRFD / YOLOv8-Face / BlazeFace`
2. generation 线
   - 继续细化 paired/unpaired 样本权重
   - 继续验证 face crop 与整图回贴策略
3. 一致性线
   - 从轻量评估指标升级到显式 identity / perceptual consistency loss
   - 从 grouped sampling 升级到同病例双视角联合损失
4. 条件增强线
   - 重新评估 `ControlNet-Depth`
   - 评估病例级 reference 方案，如 `IP-Adapter`
