# 报告001：Detection 阶段实验总结

日期：2026-04-11

## 1. 当前状态与主线关系

这一阶段对应当前主线里的 detection 部分。  
它是 generation 的前置条件层，不直接负责生成结果，而是负责把“哪里该改、哪些区域稳定、哪些区域可编辑”先定义清楚。

当前这一阶段处于：

1. 已跑通主链
2. 仍在继续优化检测质量和多视角稳定性

## 2. 阶段目标

这一阶段要解决的问题不是“生成术后图”，而是：

1. 六视角下怎么稳定检测到脸
2. 鼻子、嘴巴、眼部这些局部区域怎么稳定定位
3. 怎么把这些定位结果变成 generation 可直接消费的条件

换句话说，detection 阶段的目标是把原始图片变成结构化编辑条件。

## 3. 当前执行逻辑

当前 detection 的主流程分成 3 条：

1. `detect_compare`
   - 用多检测器对同一批数据做横向比较
   - 主要看漏检、偏移、速度和可视化效果

2. `eye_mask`
   - 生成打码后的主输入图
   - 这些图后续会被 generation 默认当作训练和推理主输入

3. `sam_mask`
   - 用检测框和关键点去构建 SAM 提示
   - 产出标准化的：
     - `face_mask`
     - `inpaint_mask`
     - `feather_mask`

这 3 条流程的共同目标，是把 detection 从“检测脚本”变成“条件构建系统”。

## 4. 主要方法与设计思路

### 4.1 为什么要做多检测器比较

因为当前数据是固定 1 到 6 号多视角图，不同角度下的难点并不一样：

1. 正脸更容易成功
2. 大侧脸和底视图更容易漏检
3. 一些模型虽然能框出脸，但鼻部或部位语义会偏

所以当前 detection 不是只看“有没有框到脸”，而是同时比较：

1. 漏检情况
2. 鼻部和部位框偏移
3. 六视角适应性
4. 速度

### 4.2 为什么 `eye_mask` 和 `sam_mask` 要分开

因为它们服务的对象不同：

1. `eye_mask`
   - 主要给 generation 提供主输入图
2. `sam_mask`
   - 主要给 generation 提供编辑区域和回贴条件

拆开以后，后续 generation 才能稳定地消费：

1. 打码图
2. `face_mask`
3. `inpaint_mask`
4. `feather_mask`

### 4.3 为什么要做视角微调

因为同一套部位框参数在不同视角下不一定成立。  
当前项目已经把视角微调统一收进配置里，允许按方法、按视角调整：

1. 定位点偏移
2. 框尺寸缩放

这一步的意义，是把“靠经验人工修”收口成“显式可维护配置”。

## 5. 已做尝试

这一阶段不是一开始就做成现在这样，而是一步步演化出来的。

### 5.1 早期尝试

1. 单脚本黑条遮眼
   - 快速
   - 但侧脸和底视图问题明显

2. MediaPipe 路线
   - 关键点细
   - 但入口检测在大侧脸下不稳定

3. MTCNN fallback
   - 能补一部分漏检
   - 但偏移和框质量问题较多

### 5.2 多检测器对比阶段

主要尝试过：

1. `MTCNN`
2. `RetinaFace`
3. `SCRFD`
4. `BlazeFace`
5. `MediaPipe Face Mesh / FaceLandmarker`
6. `YOLOv8-Face`
7. `CenterFace`

### 5.3 当前保留的方向

当前更偏向保留：

1. `SCRFD`
2. `BlazeFace`
3. `YOLOv8-Face`

因为这几个路线在 detection 成功率、速度和后续 SAM 条件适配上更实用。

## 6. 当前效果与阶段性结论

当前 detection 阶段已经达到的效果是：

1. 可以稳定输出给 generation 使用的主输入图和编辑条件
2. 多检测器比较不再是零散脚本，而是统一流程
3. 视角微调已经成为显式配置，而不是散落在代码里的启发式逻辑

当前阶段性判断是：

1. `SCRFD` 更偏稳定优先
2. `BlazeFace` 更偏速度优先
3. `YOLOv8-Face` 是折中路线

## 7. 当前局限

1. 大侧脸和底视图下，鼻部和部位框仍然可能偏移
2. 一些模型能检测到脸，但对后续 SAM 提示并不友好
3. 目前效果判断仍然有人工观察成分，量化还不够完整

## 8. 这阶段对项目意味着什么

这阶段的意义，不是“检测脚本写完了”，而是：

1. generation 的输入条件终于有了统一定义
2. 训练、推理、评估都可以围绕同一套 detection 产物来组织
3. 后续 generation 的问题可以和 detection 的问题分层看待

## 9. 后续可能思路

1. 继续比较 `SCRFD / YOLOv8-Face / BlazeFace`
2. 继续优化大侧脸和底视图下的鼻部偏移
3. 继续收紧视角微调配置
4. 让 detection 输出更稳定地服务 generation 主链
